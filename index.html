Why does APT not use HTTPS?

"Isn't it more secure…?"

tl;dr

HTTPS is used to prevent intruders from being able to listen to communications between you and websites you visit, as well as to avoid data being modified without your knowledge. It is an important tool to prevent MitM attacks5 and should be used to protect APT repositories.

Repository metadata obtained by APT can be accompanied by GPG signatures which allows your system to check that the repository was generated by a source authorized by your distribution.

These signatures are checked against a small set of trusted keys already stored on your computer. Downloaded repository metadata is rejected by APT if it is signed by an unknown key1 or are missing valid signatures. This ensures that the repository you are installing packages from is authorised by your distribution and has not been modified or replaced since.

It is important to note that APT does not verify GPG signatures on packages themselves, thus GPG signing Debian packages will have no meaningful effect for users automatically installing packages with aptitude, apt-get, or similar tools.

HTTPS can not detect if malicious tampering has occurred on the disks of the server you are downloading from. There is little point "securely" transfering a compromised package.

But what about privacy?

HTTPS does not provide meaningful privacy for obtaining packages. As an eavesdropper can usually see which hosts you are contacting, if you connect to your distribution's mirror network it would be fairly obvious that you are downloading updates.

Furthermore, even over an encrypted connection it is not difficult to figure out which files you are downloading based on the size of the transfer2.

HTTPS does provide protection from replay attacks, which are a serious concern for GPG signed APT repositories.

HTTPS should be provided in addition to GPG signed repository metadata

Your distribution should cryptographically sign the files using the existing scheme and additionally serve the files over HTTPS to provide "defense in depth."

Providing a huge worldwide mirror network available over SSL is a complicated engineering task, but prevents several MitM attacks that GPG signed APT repositories are vulnerable to and ensures that users of open source software can safely and securely install packages and updates.

Ah, what about replay attacks?

One issue with GPG signing APT repositories is that it does not guarantee that you are seeing the most up-to-date version of the archive.

This can lead to a replay attack where an attacker substitutes an archive with an earlier—unmodified—version of the archive. This would prevent APT from noticing new security updates which they could then exploit.

To mitigate this problem, APT archives includes a timestamp after which all the files are considered stale4, which limits (but does not prevent) replay attacks. The best way to protect against replay attacks is to always serve APT repositories over HTTPS.

Where can I find out more?

More technical details about GPG signed APT repositories may be found on the SecureAPT wiki page. More technical details about the various attacks that GPG signed APT repositories can fall victim to are available in this helpful blog post.
